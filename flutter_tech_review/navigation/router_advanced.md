# Flutter 路由与导航进阶

## 一、为什么导航体系会被问深

Flutter 初级导航并不复杂，很多人只会：

- `Navigator.push`
- `Navigator.pop`
- `pushNamed`

但中高级面试和真实项目更关注的是：

1. Navigator 1.0 和 2.0 的区别
2. 声明式路由为什么出现
3. Web URL 同步为什么需要 Router
4. 深链路怎么接入
5. `go_router` 为什么流行
6. 嵌套路由、认证拦截、404 页面怎么处理

## 二、Navigator 1.0 是什么

Navigator 1.0 可以理解为“命令式导航”。

你通过命令直接操作导航栈：

- push
- pop
- pushReplacement
- pushAndRemoveUntil

### 优点

- 简单直接
- 小中型页面流转足够好用
- 上手成本低

### 缺点

- URL 与页面状态同步能力弱
- Web 支持不够自然
- 导航状态不容易声明式管理
- 深链路和复杂跳转链维护成本高

## 三、为什么会有 Navigator 2.0

Navigator 2.0 的核心目标不是“API 更复杂”，而是：

- 让路由状态也能声明式表达
- 让页面栈可以由状态推导出来
- 更自然地支持浏览器地址栏、前进后退、深链路

也就是说，它想解决的是“导航状态可建模”。

## 四、Navigator 1.0 和 2.0 的本质区别

### Navigator 1.0

- 你手动操作栈
- 栈变化是命令驱动
- 更像 imperative API

### Navigator 2.0

- 你描述当前应该有哪些页面
- 页面栈由状态决定
- 更像 declarative API

### 一句话理解

1.0 是“我现在 push 一个页面”。
2.0 是“根据当前状态，页面栈应该长成这样”。

## 五、Router 体系包含什么

Flutter Router 体系主要涉及：

1. `RouteInformationProvider`
2. `RouteInformationParser`
3. `RouterDelegate`
4. `BackButtonDispatcher`

### `RouteInformationParser`

负责把 URL 或路由信息解析成应用内部状态。

### `RouterDelegate`

负责根据应用状态返回页面栈。

### `BackButtonDispatcher`

负责处理系统返回行为。

## 六、为什么很多人觉得 Router 2.0 难

不是因为概念本身不可理解，而是因为它把“导航”拆成了几个职责：

- URL 解析
- 状态建模
- 页面栈构建
- 返回行为处理

这套设计是合理的，但对只做普通移动端命令式跳转的人来说，抽象层明显更高。

## 七、什么时候该用 Navigator 1.0

适合：

- 纯移动端 App
- 页面层级不复杂
- 没有复杂 URL 需求
- 深链路较少

如果项目很简单，强行上 Router 2.0 只会增加负担。

## 八、什么时候该用 Router / Navigator 2.0

适合：

- Flutter Web
- 深链路要求高
- 希望 URL 和页面状态严格同步
- 页面权限、重定向、嵌套路由较复杂

## 九、`go_router` 为什么流行

`go_router` 流行的原因不是因为它“新”，而是因为它在 Flutter Router 体系上做了更高层封装。

它帮你简化了：

- 路由声明
- 参数解析
- 嵌套路由
- redirect
- shell route
- Web URL 同步

所以很多团队不会直接裸写 Router 2.0，而是使用 `go_router`。

## 十、`go_router` 的核心价值

### 1. 声明更集中

路由表统一定义，更适合中大型项目管理。

### 2. Web 友好

地址栏和页面状态同步更自然。

### 3. redirect 机制清晰

登录拦截、权限跳转、首次启动引导更容易统一实现。

### 4. 嵌套路由更实用

例如底部 Tab + 各自独立导航栈，这类场景更容易组织。

## 十一、动态参数和查询参数

导航里高频会问两类参数：

### 路径参数

例如：

`/product/123`

这种参数更适合表达资源身份。

### 查询参数

例如：

`/search?keyword=flutter&page=2`

这种参数更适合表达筛选条件、分页、排序等可选信息。

### 面试回答不要混淆

路径参数强调资源定位；
查询参数强调附加条件。

## 十二、深链路是什么

深链路就是从应用外部直接打开应用内部某个具体页面的能力。

入口可能来自：

- 浏览器
- 短信
- 推送通知
- 第三方 App
- 二维码

### 深链路真正难点

不是“能不能打开”，而是：

1. 冷启动和热启动如何统一处理
2. 用户未登录时怎么跳转
3. 参数无效时怎么兜底
4. 页面栈如何恢复到合理状态

## 十三、深链路接入时的工程问题

### 1. 登录拦截

如果用户未登录，通常不能直接进入目标页。

正确做法往往是：

1. 先记录目标路由
2. 跳登录
3. 登录成功后再恢复目标路由

### 2. 参数校验

不能假设外部传入参数一定合法。

需要处理：

- 缺失参数
- 类型错误
- 资源不存在

### 3. 404 页面

复杂路由体系必须有兜底页，而不是解析失败直接崩。

## 十四、嵌套路由为什么重要

在真实 App 中，经常有这种结构：

- 底部 Tab
- 每个 Tab 自己维护一个导航栈
- Tab 切换时要保留各自页面状态

这就是嵌套路由的典型场景。

如果只用最简单的 push/pop，很容易把各 Tab 的导航历史搅在一起。

## 十五、认证和权限重定向

中大型项目路由系统里，一个高频问题是：

“用户有没有资格访问这个页面？”

常见策略：

1. 未登录跳登录页
2. 登录后跳回目标页
3. 无权限进入 403 或降级页
4. 参数非法进入 404 页

这类逻辑最适合集中在路由层做，不要散落在每个页面里。

## 十六、导航和状态管理的关系

一个成熟项目里，导航状态往往不是完全孤立的。

它常常和这些状态相关：

- 登录态
- 权限态
- 首次启动引导状态
- 选中的 Tab
- 当前业务上下文

所以导航体系做深之后，通常会和状态管理方案耦合。

## 十七、面试高频问答

### 1. Navigator 1.0 和 2.0 的区别是什么？

1.0 是命令式导航，手动操作路由栈；2.0 是声明式导航，根据状态描述页面栈，更适合 Web、深链路和复杂路由管理。

### 2. 为什么 Flutter Web 更需要 Router 2.0？

因为 Web 要处理地址栏、浏览器前进后退、URL 与页面状态同步，这些都更适合声明式路由模型。

### 3. `go_router` 的优势是什么？

它对 Router 2.0 做了更高层封装，简化了声明式路由、重定向、嵌套路由、参数处理和 Web URL 同步。

### 4. 深链路最难的点是什么？

不是跳进去本身，而是冷启动、登录拦截、参数校验、页面栈恢复和异常兜底。

### 5. 为什么复杂项目不建议把跳转逻辑散落在页面里？

因为认证、权限、重定向、404、参数校验等逻辑一旦分散，就会难维护、难排查、难统一修改。

## 十八、推荐回答模板

如果面试官问“你如何设计 Flutter 路由体系”，更完整的回答方式是：

1. 小型移动端项目可以先用 Navigator 1.0
2. 如果涉及 Web、深链路、认证拦截、嵌套路由，优先考虑声明式路由
3. 实战中通常会选择 `go_router` 这类封装，而不是手写整套 Router 2.0
4. 路由层应集中处理参数校验、权限控制、404 和重定向

这样回答，比单纯说“我用 pushNamed”成熟得多。
