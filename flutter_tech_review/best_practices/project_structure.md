# Flutter 项目结构设计

## 一、为什么项目结构会成为中高级考点

Flutter 小项目往往靠页面堆起来就能跑，但项目一旦变大，真正决定维护成本的往往不是控件，而是结构。

面试问项目结构，本质是在问：

1. 你如何控制复杂度
2. 你如何划分模块边界
3. 你如何让多人协作不失控
4. 你如何让测试、依赖、路由、状态管理可落地

## 二、项目结构设计的目标

好的结构不是“目录好看”，而是要解决这些问题：

1. 代码好找
2. 职责清晰
3. 可复用
4. 可测试
5. 便于扩展

## 三、常见三种组织方式

### 1. 按类型分层

例如：

- `pages/`
- `widgets/`
- `models/`
- `services/`

### 优点

- 刚开始最直观
- 小项目上手快

### 缺点

- 业务扩大后跨目录跳转频繁
- 同一个功能的代码分散
- 模块边界弱

## 四、feature-first 结构

这种方式按业务功能划分目录，例如：

- `features/auth/`
- `features/home/`
- `features/profile/`

每个 feature 下再放自己的：

- page
- state
- repository
- widget

### 优点

- 功能聚合度高
- 更适合多人协作
- 改动范围更清晰

### 缺点

- 共享基础设施需要额外抽离
- 如果没有规范，feature 内部也可能继续膨胀

## 五、layer-first 结构

这种方式按职责分层，例如：

- presentation
- domain
- data

### 优点

- 结构清晰
- 适合强调架构约束的项目
- 测试边界明确

### 缺点

- 小项目容易显重
- 初学者容易写成“伪分层”

## 六、clean architecture 在 Flutter 里的落地

Flutter 中常见的 clean architecture 一般分为三层：

### 1. presentation

包含：

- page
- widget
- state notifier / bloc / controller

### 2. domain

包含：

- entity
- use case
- repository contract

### 3. data

包含：

- repository impl
- remote datasource
- local datasource
- dto / model

### 关键价值

- 把业务规则和 UI 分开
- 把数据实现和抽象分开

## 七、不要机械上 clean architecture

这是高频误区。

不是所有 Flutter 项目都值得完整上三层 clean architecture。

如果项目：

- 规模小
- 生命周期短
- 团队人数少
- 业务简单

那么过重的分层会拖慢开发效率。

更合理的做法是：

- 小项目优先简单
- 中大型项目再逐步增加边界约束

## 八、一个更实用的推荐结构

很多 Flutter 中大型项目适合：

1. `core/`
2. `shared/`
3. `features/`

### `core/`

放全局基础设施：

- 路由
- 网络客户端
- 错误体系
- 依赖注入
- 主题
- 日志

### `shared/`

放跨业务共享但不属于核心基础设施的内容：

- 通用组件
- 通用模型
- 工具方法
- 扩展方法

### `features/`

按业务功能拆分模块。

## 九、模块边界怎么划

这比目录名字更重要。

一个稳定模块通常要明确：

1. 它暴露什么能力
2. 它依赖什么外部能力
3. 它的状态边界在哪里
4. 它的数据从哪里来

### 错误做法

- 任意模块直接访问任意实现层
- UI 直接调用底层数据源
- 跨模块互相引用实现细节

## 十、页面、组件、状态、数据的推荐边界

### 页面层

负责：

- 组装 UI
- 响应用户交互
- 订阅状态

### 状态层

负责：

- 状态建模
- 业务流程驱动
- 暴露页面可消费状态

### 数据层

负责：

- 请求接口
- 读写本地缓存
- 数据转换

### 原则

不要让页面层直接承担业务编排和数据落地。

## 十一、命名和目录规范

结构再好，如果命名失控，也会迅速失效。

建议：

1. 目录名和业务能力对应
2. 文件名表达职责，不表达临时实现细节
3. 同一层命名保持一致

例如不要同时出现：

- `user_service.dart`
- `user_repo.dart`
- `user_manager.dart`

但它们职责又说不清。

## 十二、面试高频问答

### 1. Flutter 项目结构你会怎么设计？

我会先看项目规模和团队复杂度。小项目可以简单分层，中大型项目更适合 `core + shared + features` 或 feature-first 结构，必要时在 feature 内做 presentation/domain/data 边界划分。

### 2. 为什么不建议只按 `pages/widgets/services` 划分？

因为随着业务增长，同一功能的代码会被分散到多个目录，维护和协作成本会持续上升。

### 3. clean architecture 一定要上吗？

不一定。它适合复杂项目，但对小项目可能过重，关键是根据规模控制复杂度，而不是机械套模板。

### 4. 模块边界最重要的是什么？

不是目录名字，而是职责、依赖方向和对外暴露接口是否清晰。

## 十三、推荐回答模板

如果面试官问“Flutter 项目结构怎么设计”，比较成熟的回答是：

1. 先根据项目规模选择简单结构或 feature-first
2. 中大型项目优先按业务模块聚合
3. 把网络、路由、错误处理、依赖注入沉到 `core`
4. 在复杂业务模块内部再做 presentation / domain / data 边界划分

这样回答比直接说“我喜欢 clean architecture”更稳。
