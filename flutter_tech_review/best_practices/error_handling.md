# Flutter 错误处理与崩溃治理

## 一、为什么错误处理是工程能力分水岭

很多 Flutter 项目能跑，但一出线上问题就很难排查，核心原因通常不是“框架不行”，而是错误处理体系没建立起来。

面试问错误处理，本质在问：

1. 你怎么分层定义错误
2. 你怎么收集错误
3. 你怎么避免 UI 到处写 `try/catch`
4. 你怎么让线上问题可定位

## 二、错误不应该只分“有错”和“没错”

一个成熟项目至少要区分：

1. 网络错误
2. 业务错误
3. 解析错误
4. 平台错误
5. 未知错误

如果所有异常最后都变成“请求失败”，排查成本会很高。

## 三、推荐的错误分层

### 1. 数据层错误

例如：

- HTTP 超时
- 状态码异常
- JSON 解析失败
- 本地存储失败

### 2. 领域层错误

例如：

- 用户无权限
- 状态不合法
- 业务规则不满足

### 3. 表现层错误

例如：

- Toast
- 错误页
- 弹窗提示

### 原则

UI 只负责展示错误，不负责定义底层错误语义。

## 四、为什么不能到处裸写 `try/catch`

因为那会导致：

1. 处理逻辑分散
2. 错误文案不统一
3. 日志和上报漏掉
4. 页面层越来越重

更合理的做法是：

- 数据层把底层异常转换成统一错误模型
- 状态层决定业务如何响应
- UI 只消费最终可展示状态

## 五、网络错误应该怎么处理

网络错误不要只看“有没有异常”，还要区分：

1. 连接失败
2. 超时
3. 服务器错误
4. 鉴权失败
5. 数据格式异常

### 推荐做法

在网络层统一拦截：

- 请求日志
- 响应日志
- 错误映射
- token 过期处理

## 六、全局错误捕获

Flutter 工程里，高频考点之一就是：

“全局错误怎么接？”

### 常见入口

1. `FlutterError.onError`
2. `PlatformDispatcher.instance.onError`
3. `runZonedGuarded`

### 各自意义

- `FlutterError.onError`：捕获 Flutter framework 层错误
- `PlatformDispatcher.instance.onError`：捕获部分未处理异步错误
- `runZonedGuarded`：提供 zone 级别兜底

### 面试不要答浅

不要只说“我会用 `FlutterError.onError`”，那是不够的。

## 七、为什么异步错误更难

因为异步错误不一定沿着同步调用栈冒泡。

你需要分别考虑：

1. `Future` 错误
2. `Stream` 错误
3. isolate 错误
4. 平台通道错误

这些都不能靠一个统一 `try/catch` 全包住。

## 八、状态管理里的错误表达

比起直接抛异常给 UI，更稳的方式是把错误建模成状态的一部分。

例如：

- loading
- success(data)
- failure(error)

这样页面就能统一处理：

- loading 态
- 空态
- 错误态
- 重试态

## 九、崩溃上报应该包含什么

崩溃上报不只是把 stack trace 发出去。

更有价值的信息包括：

1. 用户标识
2. 应用版本
3. 页面路径
4. 关键业务上下文
5. 设备与系统信息
6. 最近操作链路

否则线上崩溃即使收到了，也不容易定位。

## 十、日志体系怎么配合错误处理

日志至少要分级：

1. debug
2. info
3. warning
4. error

### 原则

- 开发环境可以详细
- 生产环境要避免敏感信息泄露
- 错误日志要能关联请求、用户和页面

## 十一、用户提示和技术错误要分开

这是很重要的一点。

不要把底层异常原文直接展示给用户。

应当分离：

- 用户可读文案
- 技术诊断信息

例如：

- 用户看到：“网络异常，请稍后重试”
- 日志里记录：`SocketException`、请求路径、traceId

## 十二、重试和降级

一个成熟错误体系不只是“报错”，还要考虑恢复手段。

### 重试适合

- 短时网络波动
- 幂等请求失败

### 不适合盲目重试

- 参数错误
- 权限错误
- 业务校验失败

### 降级场景

- 缓存兜底
- 展示旧数据
- 功能关闭开关

## 十三、Platform Channel 和 FFI 的错误边界

### Platform Channel

错误通常可以转成可处理的异常或错误码。

### FFI

风险更高。

因为：

- 签名不匹配可能直接崩溃
- 非法内存访问不会像 Dart 异常一样温和可控

所以 FFI 侧更需要：

- 边界检查
- 错误码协议
- 原生层兜底

## 十四、面试高频问答

### 1. Flutter 如何做全局错误捕获？

通常组合使用 `FlutterError.onError`、`PlatformDispatcher.instance.onError` 和 `runZonedGuarded`，分别兜住 framework、异步和 zone 层错误。

### 2. 为什么不建议在页面里到处写 `try/catch`？

因为会导致错误处理分散、提示不统一、日志上报缺失，应该由数据层和状态层统一收敛错误语义。

### 3. 网络错误和业务错误为什么要区分？

因为它们的恢复方式和用户提示完全不同，混在一起会让系统难维护。

### 4. 错误处理体系的核心目标是什么？

不是“别报错”，而是错误可感知、可定位、可恢复、可治理。

## 十五、推荐回答模板

如果面试官问“Flutter 项目如何做错误处理”，比较完整的回答方式是：

1. 先按网络、业务、解析、平台、未知错误做分层
2. 在数据层统一映射底层异常
3. 在状态层把错误表达为可消费状态
4. UI 只负责展示和重试
5. 再用全局捕获和崩溃上报补齐兜底链路

这样回答会明显强于“我会 try/catch 和上报 crash”。
