# Flutter 依赖注入与模块装配

## 一、为什么依赖注入会被问

Flutter 项目一旦变大，就会遇到这些问题：

1. 对象在哪里创建
2. 谁负责生命周期
3. 如何替换 mock
4. 不同模块如何拿到同一服务

依赖注入要解决的不是“少写 new”，而是对象装配和边界管理问题。

## 二、什么是依赖注入

依赖注入的核心是：

- 一个对象不自己创建依赖
- 依赖从外部提供给它

例如：

- 页面不自己 new repository
- repository 不自己 new http client

这样做的好处是：

- 降低耦合
- 方便测试
- 更容易替换实现

## 三、不要把依赖注入理解成某个库

这是面试里的常见误区。

依赖注入是一种设计方式，`get_it`、`injectable`、Riverpod provider 都只是实现手段。

所以不要把回答说成：

“我用了 get_it，所以我做了依赖注入。”

更准确的是：

“我通过容器或 provider 管理对象创建和装配，实现依赖注入。”

## 四、为什么对象不要自己 new 依赖

如果对象内部自己创建依赖，会带来：

1. 难测试
2. 难替换实现
3. 依赖关系不可见
4. 生命周期难管理

例如页面内部直接：

- new Dio
- new Repository
- new Service

这种做法在小 demo 还能接受，在项目里会迅速失控。

## 五、常见依赖注入方式

### 1. 构造函数注入

最直接，也最推荐优先考虑。

优点：

- 依赖显式
- 测试友好
- 代码清晰

### 2. 容器注入

例如通过 `get_it` 统一注册和获取对象。

优点：

- 全局装配方便
- 适合工程化组织

缺点：

- 容易变成隐式依赖

### 3. Provider / Riverpod 注入

通过状态和依赖系统统一管理对象来源。

适合：

- 与状态管理强结合
- 页面级和模块级依赖组织

## 六、`get_it` 的定位

`get_it` 更像是 service locator 容器。

它解决的问题是：

- 对象统一注册
- 按需获取
- 生命周期管理

### 适合场景

- app 级单例
- repository / service 装配
- 配合模块初始化

### 风险

如果滥用，会让依赖变得过于隐式。

所以最好仍然在业务对象内部使用构造函数接收依赖，而不是到处直接 `getIt()`。

## 七、`injectable` 的价值

`injectable` 的核心价值是帮你做自动注册代码生成。

适合：

- 项目依赖较多
- 手写注册表太长
- 需要统一生命周期声明

但要注意：

它提升的是工程效率，不会自动帮你设计出好边界。

## 八、生命周期怎么管理

依赖注入里，高频面试点是生命周期。

常见类型：

1. singleton
2. lazy singleton
3. factory

### singleton

应用生命周期内通常只有一个实例。

适合：

- 日志
- 配置
- 网络客户端

### lazy singleton

第一次使用时创建。

适合：

- 初始化成本较高但未必会用到的服务

### factory

每次获取都新建。

适合：

- 短生命周期对象
- 不应该共享状态的对象

## 九、模块化装配怎么做

当项目分 feature 后，依赖注册最好也按模块组织。

例如：

- auth module register
- home module register
- profile module register

### 优点

- 依赖边界更清晰
- 模块迁移更容易
- 初始化过程可控

## 十、测试为什么依赖注入很重要

没有依赖注入时，测试常见问题是：

- 真请求被发出去
- 真数据库被读写
- 平台能力难 mock

有依赖注入后，你可以：

- 替换 repository
- 替换 service
- 注入 fake / mock

这也是面试里很加分的一点。

## 十一、依赖注入和状态管理的关系

很多项目里，依赖注入和状态管理会配合出现。

例如：

- Riverpod provider 既提供状态，也提供依赖
- Bloc 需要从外部拿 use case / repository
- 页面状态对象不应该自己 new 数据层对象

### 原则

状态管理解决“状态怎么流动”，依赖注入解决“对象怎么装配”。

不要把两者混成一个问题。

## 十二、常见误区

### 1. 全部做成单例

这会让状态和生命周期混乱。

### 2. 到处直接访问全局容器

这样虽然方便，但依赖会越来越隐式。

### 3. 只用库，不做边界设计

即使上了 `get_it` / `injectable`，如果模块边界不清晰，项目仍然会乱。

### 4. 页面直接创建底层依赖

这会破坏测试性和分层。

## 十三、面试高频问答

### 1. Flutter 为什么需要依赖注入？

因为随着项目变大，需要统一管理对象创建、生命周期、模块装配和测试替换，避免高耦合。

### 2. `get_it` 是不是依赖注入本身？

不是，它是实现依赖管理和对象定位的一种工具，依赖注入本质上是一种设计方式。

### 3. 为什么不建议到处直接从容器里取对象？

因为会让依赖关系变隐式，降低可读性和可测试性。

### 4. 构造函数注入和 service locator 怎么选？

优先保证依赖显式。容器适合做统一装配，但业务对象内部最好仍通过构造函数接收依赖。

## 十四、推荐回答模板

如果面试官问“Flutter 项目如何做依赖注入”，比较稳的回答方式是：

1. 先用构造函数注入保证依赖显式
2. 用容器管理 app 级对象装配和生命周期
3. 按模块拆注册逻辑
4. 测试时替换 repository / service 实现

这样回答，比单纯说“我用了 get_it”更完整。
