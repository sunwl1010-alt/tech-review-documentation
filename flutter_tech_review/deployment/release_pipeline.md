# Flutter 发布流水线：CI/CD、签名与分发

## 一、为什么发布流水线重要

如果发布完全靠手工操作，项目一大就会出现：

1. 构建不一致
2. 版本号混乱
3. 签名管理混乱
4. 人工操作容易出错

所以发布流水线的目标不是“自动化看起来高级”，而是稳定和可追踪。

## 二、发布流水线通常包括什么

一个完整的 Flutter 发布流水线通常包含：

1. 代码检查
2. 测试
3. 构建
4. 签名
5. 产物归档
6. 分发 / 上传
7. 通知

## 三、CI 和 CD 的区别

### CI

持续集成，重点是：

- 代码合并前验证质量
- 跑 analyze 和测试
- 保证主干稳定

### CD

持续交付 / 持续部署，重点是：

- 自动构建产物
- 自动签名
- 自动上传测试渠道或应用商店

## 四、Flutter 发布流水线建议拆层

### PR 流水线

跑：

- `flutter analyze`
- unit / widget test
- 必要时 golden test

### release 流水线

跑：

- release build
- 签名
- 生成 AAB / IPA / Web 产物
- 上传分发平台

### 原因

不要把高成本发布动作放到每个普通提交里。

## 五、签名和密钥管理

这是发布流水线里最敏感的部分。

### 原则

1. 不把密钥直接提交进仓库
2. 使用 CI secret 管理
3. 签名文件和密码分离管理
4. 做访问权限控制

### Android

通常涉及：

- keystore
- alias
- store password
- key password

### iOS

通常涉及：

- 证书
- provisioning profile
- App Store Connect 凭据

## 六、版本号自动化

一个成熟流水线通常不会靠人工手改每次构建号。

### 推荐做法

1. 对外版本号按发布计划维护
2. 构建号在流水线中自动递增

### 好处

- 避免重复构建号
- 便于回溯构建来源
- 方便和 Git 提交建立映射

## 七、产物管理

构建完成后，产物不应只存在 CI 临时目录里。

建议归档：

1. AAB / APK
2. IPA
3. Web build
4. 构建日志
5. 版本元数据

### 原因

这样回溯问题、回滚版本、做测试分发都会更容易。

## 八、测试分发和正式分发要分开

这是工程上很重要的一点。

### 测试分发

例如：

- Android 内测
- TestFlight
- Web staging

### 正式分发

例如：

- Google Play production
- App Store 正式发布
- 生产环境 Web

### 原则

不要让“测试构建”和“正式发布”走完全一样的权限链路。

## 九、发布失败怎么处理

成熟流水线不仅要会发，还要会失败得清楚。

至少要做到：

1. 构建失败可定位
2. 上传失败可重试
3. 通知明确
4. 产物可回溯

## 十、回滚策略

很多团队只想“怎么发”，不想“怎么回滚”。

### 移动端

移动端回滚通常没有 Web 那么直接，因此更依赖：

- 灰度
- 审核前内测
- 远程配置兜底

### Web

Web 回滚通常更快，但要注意：

- 资源缓存
- CDN 刷新
- 版本一致性

## 十一、环境变量和配置管理

发布流水线常常要区分：

- API 地址
- 功能开关
- 日志等级
- 第三方 key

### 原则

环境配置要和代码、签名、发布目标一起被显式管理，不能靠手工临时修改。

## 十二、面试高频问答

### 1. Flutter 项目如何做 CI/CD？

通常会把 analyze、测试、构建、签名、产物归档和上传分成不同阶段，由 CI/CD 流水线自动完成。

### 2. 为什么签名信息不能直接放仓库？

因为它涉及发布身份和供应链安全，一旦泄露，风险非常高。

### 3. 为什么测试分发和正式分发要分开？

因为两者的权限、风险和审核要求不同，混在一起会放大误发布风险。

### 4. 发布流水线最核心的价值是什么？

稳定、一致、可回溯，而不是单纯“少点几下按钮”。

## 十三、推荐回答模板

如果面试官问“Flutter 发布流程怎么做自动化”，比较成熟的回答方式是：

1. 先在 PR 流程里完成 analyze 和测试
2. release 流水线再做 release build、签名和产物归档
3. 使用 secret 管理签名信息和商店凭据
4. 把测试分发、正式发布、回滚策略和版本号管理纳入同一套流程

这样回答会比“我用 GitHub Actions 打包”更完整。
