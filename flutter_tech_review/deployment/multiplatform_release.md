# Flutter 多平台发布进阶

## 一、为什么发布体系会被问深

Flutter 的优势之一是多平台，但“能跑”不等于“能发”。

真正进入项目阶段后，发布要面对的是：

1. Android / iOS / Web / Desktop 的差异
2. 签名与证书
3. 包体积与构建产物
4. 渠道和商店规则
5. 灰度、测试分发、回滚

## 二、四类平台的发布目标不同

### Android

核心关注：

- AAB / APK
- 签名
- 渠道
- Google Play 审核与分发

### iOS

核心关注：

- 证书与 provisioning profile
- Archive / IPA
- TestFlight
- App Store Connect 审核

### Web

核心关注：

- 静态资源部署
- CDN / HTTPS
- 路由与缓存策略
- SEO / PWA

### Desktop

核心关注：

- 平台打包格式
- 安装器
- 签名与 notarization
- 分发渠道

## 三、Android 发布要点

### 1. 为什么现在更推荐 AAB

因为 Google Play 更倾向使用 App Bundle，由商店按设备架构和资源拆分下发，更节省用户下载体积。

### 2. 签名的意义

Android 签名不仅是“能装”，还是：

- 应用身份标识
- 更新一致性保障
- 分发链安全基础

### 3. 常见构建变体

- debug
- release
- flavor

中大型项目通常会通过 flavor 区分：

- dev
- staging
- prod

## 四、iOS 发布要点

### 1. iOS 的核心难点不在 Flutter

而在 Apple 生态本身：

- 证书
- 描述文件
- 签名链
- 审核规则

### 2. TestFlight 的价值

它不是正式发布，但对团队非常重要：

- 内测
- 回归验证
- 提前发现审核问题

### 3. 常见风险

- 签名不一致
- capability 配置缺失
- 隐私说明不完整
- 审核元数据不全

## 五、Web 发布要点

### 1. Web 不是“build 完就结束”

Web 发布至少还要考虑：

1. 静态资源路径
2. 路由刷新 404
3. CDN 缓存
4. HTTPS
5. SEO 和首屏体验

### 2. 路由部署问题

如果使用 history 模式，服务端必须正确回退到入口页面，否则用户刷新深链路会 404。

### 3. 缓存策略

缓存如果设计不好，会出现：

- 新版本发布后用户拿到旧 bundle
- 静态资源 hash 不一致

## 六、Desktop 发布要点

Desktop 的难点通常不在代码，而在分发形态。

### 常见问题

1. Windows 安装器
2. macOS notarization
3. Linux 打包格式
4. 自动更新机制

### 关键结论

Desktop 不是简单 `flutter build windows` 就完事，还要面对安装、签名、升级体验。

## 七、版本管理怎么做

Flutter 发布里至少要分清：

1. 用户看到的版本号
2. 构建号

### 为什么要分

因为：

- 用户看到的是语义版本
- 商店和系统判断更新常依赖构建号

### 推荐思路

- 语义版本控制对外版本
- 构建号在流水线中自动递增

## 八、灰度与多环境

中大型项目通常不会只有一个环境。

常见环境：

- dev
- test
- staging
- prod

### 为什么要做环境隔离

因为：

- API 地址不同
- 日志策略不同
- 功能开关不同
- 测试账号和正式账号隔离

## 九、包体积和发布体验

发布不仅要能过审核，还要考虑用户下载和安装体验。

### 常见优化点

1. 资源压缩
2. 去掉无用依赖
3. tree shaking
4. 按平台拆分资源

### 面试里不要答浅

不要只说“压图片”，还要知道：

- Android AAB 的按需下发
- Web 静态资源缓存
- Desktop 安装包体积对分发影响

## 十、商店审核为什么经常出问题

因为发布问题不只来自代码。

常见原因：

1. 权限说明不完整
2. 隐私政策缺失
3. 元数据不准确
4. 截图和功能描述不匹配
5. 第三方 SDK 合规问题

## 十一、面试高频问答

### 1. Flutter 多平台发布最大的挑战是什么？

不是代码复用本身，而是各平台的签名、分发、审核规则和构建产物差异。

### 2. Android 为什么更推荐 AAB？

因为它更适合商店按设备分发资源，能降低最终安装体积。

### 3. Flutter Web 发布需要特别注意什么？

服务端路由回退、缓存策略、HTTPS 和静态资源部署。

### 4. iOS 发布最容易卡在哪？

通常卡在证书、配置文件、能力配置和审核元数据，而不是 Flutter 代码本身。

## 十二、推荐回答模板

如果面试官问“Flutter 多平台发布怎么做”，更完整的回答方式是：

1. Android、iOS、Web、Desktop 分别看待，不把它们当同一种发布问题
2. 移动端重点处理签名、版本号、商店审核和分环境
3. Web 重点处理静态部署、路由回退和缓存
4. Desktop 重点处理安装器、签名和升级分发

这样回答会比“执行 build 命令再上传”成熟得多。
