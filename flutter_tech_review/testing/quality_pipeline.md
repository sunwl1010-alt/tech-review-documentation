# Flutter 质量保障：CI、覆盖率与质量门禁

## 一、为什么测试不等于质量体系

很多项目有测试，但质量依然不稳，原因是：

- 测试没有接入 CI
- 没有 lint 和格式检查
- 没有覆盖率监控
- 没有质量门禁

所以测试只是质量体系的一部分。

## 二、质量保障的核心链路

一个更完整的 Flutter 质量链路通常包括：

1. 静态检查
2. 单元测试
3. Widget 测试
4. 集成测试
5. 覆盖率统计
6. 构建验证
7. 发布前门禁

## 三、静态检查应该包含什么

至少要有：

1. `flutter analyze`
2. 格式检查
3. 规则约束

### 为什么重要

因为很多问题根本不该等到运行时或测试阶段才发现。

## 四、测试在 CI 里的推荐分层

### PR 阶段

建议跑：

1. analyze
2. unit test
3. widget test

### 主干或 nightly

建议补：

1. integration test
2. golden test
3. 覆盖率汇总

### 原因

集成测试和视觉回归通常更慢，不一定适合每次提交都全量执行。

## 五、覆盖率怎么看才合理

覆盖率有价值，但不能迷信数字。

### 误区

- 只追求 100%
- 测了 getter/setter 就当高质量

### 更合理的理解

覆盖率只能说明“跑到了多少代码”，不能直接说明“测得好不好”。

### 更该关注

1. 核心业务路径是否被覆盖
2. 错误分支是否被覆盖
3. 边界条件是否被覆盖

## 六、质量门禁是什么

质量门禁就是：

不满足标准，不允许合并或发布。

### 常见门禁

1. analyze 不能报错
2. 核心测试必须通过
3. 覆盖率不能低于阈值
4. 关键页面 golden 不能异常

## 七、为什么质量门禁很重要

因为没有门禁时，团队很容易出现：

- 先合并再补测试
- 已知失败先跳过
- 质量标准随人变化

门禁的本质是把“口头要求”变成“系统约束”。

## 八、CI 中如何接 Flutter

最常见的一条链路是：

1. 安装 Flutter
2. 获取依赖
3. 运行 analyze
4. 运行 test
5. 生成覆盖率
6. 输出报告

### 关键点

- 固定 Flutter 版本
- 缓存依赖
- 控制任务拆分

## 九、Golden Test 在 CI 中要注意什么

Golden Test 最大的问题不是写，而是环境一致性。

需要注意：

1. 字体一致
2. 分辨率一致
3. 平台环境一致
4. 渲染差异控制

否则就会出现大量无意义 diff。

## 十、为什么集成测试不宜全量依赖真实环境

因为真实环境往往：

- 不稳定
- 慢
- 成本高

更稳妥的策略是：

1. 大部分逻辑靠单元和 Widget 测试兜住
2. 少量关键链路做集成验证
3. 必要时区分 mock backend 和真实 backend 测试

## 十一、回归测试怎么做更划算

不是所有改动都需要重跑所有测试。

更实用的思路：

1. 核心逻辑全量单测
2. 高价值页面做 Widget / golden 回归
3. 关键业务链路做集成回归

### 原则

把测试资源投入在高风险、高价值区域，而不是机械平均分配。

## 十二、面试高频问答

### 1. Flutter 项目如何接 CI？

通常会把 `flutter analyze`、单元测试、Widget 测试接入 PR 流程，集成测试和覆盖率统计放在主干或夜间任务中。

### 2. 为什么覆盖率不能代表质量？

因为覆盖率只说明执行到了多少代码，不代表断言充分、边界完整或业务关键路径被正确验证。

### 3. 质量门禁有什么意义？

它把团队质量要求变成系统约束，避免“先合并、后补救”的惯性。

### 4. 为什么集成测试不能承担全部质量职责？

因为它慢、脆弱、维护成本高，应该只覆盖关键业务主流程。

## 十三、推荐回答模板

如果面试官问“Flutter 项目如何做质量保障”，比较成熟的回答方式是：

1. 先用 analyze 和 lint 做静态质量约束
2. 用测试金字塔覆盖逻辑、UI 和关键流程
3. 把测试和覆盖率接到 CI
4. 用质量门禁保证未达标代码不能随意进入主干

这样回答会比“我们有单测”更完整。
