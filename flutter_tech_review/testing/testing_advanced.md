# Flutter 测试进阶：分层、Mock、Golden

## 一、为什么测试会被问深

Flutter 面试问测试，通常不是问你会不会写 `testWidgets`，而是问：

1. 测试分层怎么做
2. 哪些逻辑该做单元测试，哪些该做 Widget 测试
3. mock 和 fake 怎么选
4. golden test 有什么价值
5. 为什么很多团队写了测试却不稳定

## 二、测试金字塔在 Flutter 里的落地

一个更稳的思路是测试金字塔：

1. 大量单元测试
2. 适量 Widget 测试
3. 少量集成测试

### 为什么这样分

因为：

- 单元测试最快、最稳定
- Widget 测试覆盖 UI 行为但成本更高
- 集成测试最接近真实，但最慢、最脆弱

## 三、单元测试适合测什么

适合：

1. 纯函数
2. 状态转换逻辑
3. use case
4. repository 逻辑
5. 数据转换

### 不适合

- 依赖复杂 UI 树的行为
- 需要真实渲染验证的内容

## 四、Widget 测试适合测什么

适合：

1. 页面状态切换
2. 按钮点击响应
3. 输入交互
4. 条件渲染
5. 错误态、空态、loading 态

### 关键价值

它验证的是“用户可见行为”，而不是内部实现细节。

## 五、集成测试适合测什么

适合：

1. 登录流程
2. 下单流程
3. 支付前主流程
4. 跨页面关键业务链路

### 不要滥用

如果把大量边界逻辑都压到集成测试，执行会慢，稳定性也会很差。

## 六、mock、fake、stub 的区别

这是高频考点。

### mock

用于验证交互和调用行为。

适合：

- 验证某方法是否被调用
- 验证参数是否正确

### fake

提供一个简化但可运行的实现。

适合：

- 内存版 repository
- 简化版本地存储

### stub

只返回预设结果，不强调交互验证。

### 实战建议

能用 fake 的场景，很多时候比大量 mock 更稳、更可读。

## 七、为什么测试容易脆弱

常见原因：

1. 依赖实现细节而不是行为
2. 依赖精确文案、布局层级、内部结构
3. 异步等待不稳定
4. 测试数据耦合严重
5. 多个测试共享状态

## 八、Golden Test 是什么

Golden Test 本质上是视觉回归测试。

它通过保存基准截图，对比后续渲染结果是否发生变化。

### 适合场景

1. 设计系统组件
2. 复杂 UI 组件
3. 多状态组件
4. 高价值页面局部区域

### 不适合滥用

如果对所有页面都上 golden，维护成本会非常高。

## 九、Golden Test 的真实价值

它最有价值的地方不是“截图”，而是：

- 防止 UI 无意回归
- 帮助设计系统稳定演进
- 让视觉变更更可审查

## 十、异步测试为什么常出问题

Flutter 测试里大量失败都跟异步有关。

常见原因：

1. 忘记 `await`
2. `pump` 次数不够
3. `pumpAndSettle` 等不到稳定状态
4. 定时器和动画没控制好

### 关键原则

异步测试不是“多等一会儿”，而是明确知道状态什么时候应该稳定。

## 十一、状态管理如何测试

### Provider / Riverpod / Bloc

推荐思路：

1. 先测状态逻辑本身
2. 再测状态变化如何驱动 UI

### 不要把所有验证都压到 Widget 测试

否则会导致：

- 测试慢
- 失败难定位
- 边界覆盖差

## 十二、网络层怎么测试

推荐拆两层：

1. repository / datasource 单元测试
2. 页面加载状态 Widget 测试

### 原则

不要在大部分测试里发真实网络请求。

除非是明确的集成验证，否则应使用 fake / mock。

## 十三、选择器和查找器怎么写更稳

Widget 测试里最好优先用：

1. 语义明确的文本
2. `Key`
3. 稳定的 widget type

少依赖：

- 复杂层级定位
- 容易变化的内部布局结构

## 十四、面试高频问答

### 1. Flutter 测试分几层？

通常分为单元测试、Widget 测试和集成测试，遵循测试金字塔原则。

### 2. mock 和 fake 怎么选？

如果要验证交互行为，用 mock；如果需要一个简化但可运行的实现，优先 fake。

### 3. Golden Test 的作用是什么？

它用于做视觉回归验证，防止 UI 无意变化，尤其适合设计系统和复杂组件。

### 4. 为什么很多 Widget 测试不稳定？

通常是因为依赖实现细节、异步等待不稳、共享状态或查找方式脆弱。

## 十五、推荐回答模板

如果面试官问“Flutter 测试怎么设计”，比较成熟的回答方式是：

1. 用单元测试覆盖大多数逻辑和状态转换
2. 用 Widget 测试覆盖关键交互与页面状态
3. 用少量集成测试覆盖主业务流程
4. 对高价值 UI 组件使用 golden test 做视觉回归

这样回答会比“我会写单测和 Widget 测试”更完整。
