# Flutter 状态管理选型

## 一、选型不是选库名，而是选状态边界

Flutter 状态管理面试最容易答偏的一点是：

大家都在讨论 Provider、Bloc、GetX、Riverpod、Redux 谁更好，但真正应该先判断的是：

1. 状态是局部还是全局
2. 状态是同步还是异步
3. 业务流是简单交互还是复杂流程
4. 团队能否长期维护这套模式

所以状态管理选型本质上是工程问题，不是偏好问题。

## 二、先分清状态类型

### 1. 局部 UI 状态

例如：

- 当前 tab
- 开关是否选中
- 弹窗显隐
- 输入框内容

这类状态通常用：

- `setState`
- `ValueNotifier`
- 局部 provider

### 2. 页面级业务状态

例如：

- 列表加载中
- 表单提交状态
- 详情页请求结果

这类状态通常适合：

- Riverpod
- Bloc
- Provider

### 3. 应用级共享状态

例如：

- 登录态
- 用户信息
- 主题
- 权限信息

这类状态才值得考虑全局化管理。

## 三、五种方案的核心差异

### Provider

优点：

- 入门简单
- 和 Flutter 生态结合自然
- 适合中小项目

缺点：

- `BuildContext` 依赖明显
- 大项目里依赖关系容易散

### Riverpod

优点：

- 不依赖 `BuildContext`
- 依赖管理清晰
- 可测试性好
- 异步状态表达更完整

缺点：

- 团队需要建立统一约定
- 初学者容易把 provider 用得过多过碎

### Bloc

优点：

- 事件到状态的边界明确
- 复杂业务流更清晰
- 团队协作时一致性强

缺点：

- 样板代码较多
- 简单场景容易显重

### GetX

优点：

- 上手快
- 开发效率高
- 路由、状态、依赖注入一体化

缺点：

- 约束弱，团队风格容易失控
- 大型项目稳定性依赖团队规范

### Redux

优点：

- 状态流严格
- 可追踪性强
- 适合复杂流程型系统

缺点：

- 模板代码多
- 心智负担大
- 对多数移动业务偏重

## 四、选型建议

### 小型项目

建议：

- `setState` + Provider
- 或 `setState` + Riverpod

重点是简单、稳定、开发快。

### 中型项目

建议：

- Riverpod
- Bloc

看业务复杂度和团队习惯决定。

### 大型项目

建议：

- Bloc 或 Riverpod

这里的关键不是“谁更高级”，而是谁更适合团队保持一致性。

### 流程驱动、审计要求高的系统

建议：

- Redux
- 或强约束 Bloc 架构

## 五、什么时候不要过度设计

以下情况不建议上重型状态管理：

1. 页面就一两个简单交互
2. 状态只在当前 Widget 内使用
3. 项目仍在快速试错阶段

因为状态管理过重会直接拖慢迭代效率。

## 六、面试高频问答

### 1. 你怎么选择状态管理方案？

我会先判断状态范围、异步复杂度、业务流复杂度和团队维护成本，再决定是局部状态方案还是全局状态方案，而不是先拍脑袋选库。

### 2. 为什么复杂场景下不能只看开发效率？

因为状态管理最终要服务的是长期维护、调试、协作和扩展，不是只服务于第一周的开发速度。

### 3. Bloc 和 Riverpod 怎么选？

如果业务流强、事件边界明确、团队希望统一范式，Bloc 更稳；如果更看重依赖注入、组合能力和灵活性，Riverpod 更合适。

### 4. 为什么很多状态不应该做成全局？

因为全局状态会增加耦合、生命周期复杂度和更新影响范围，局部状态应该尽量局部解决。

## 七、推荐答题框架

如果面试官问“你会选哪种状态管理”，建议按这个结构回答：

1. 先按状态范围拆分
2. 再看业务复杂度
3. 再看团队协作和测试需求
4. 最后给出库的选择和理由

这样回答更像工程决策，而不是个人偏好。
