# Flutter Redux 深入

## 一、Redux 的定位

Redux 不是 Flutter 专属方案，它本质上是一种可预测状态容器模式。

它的核心目标是：

- 状态集中管理
- 状态变更路径单一
- 每次变更都可追踪
- 方便调试和回放

所以 Redux 的价值不在于“代码少”，而在于“状态流严格”。

## 二、Redux 的三大核心概念

### 1. Store

Store 是全局状态容器，保存整个应用状态树。

### 2. Action

Action 是“发生了什么”的描述，不负责真正修改状态。

例如：

- 用户点击登录
- 商品加入购物车
- 请求成功返回数据

### 3. Reducer

Reducer 是纯函数，负责根据旧状态和 action 生成新状态。

关键点：

- 不直接修改旧状态
- 返回新状态
- 没有副作用

## 三、为什么 Redux 强调单向数据流

Redux 的核心链路是：

`View -> Dispatch(Action) -> Reducer -> New State -> View Update`

这样做的好处：

- 状态来源统一
- 调试链路清晰
- 方便记录和回放

代价也很明显：

- 模板代码偏多
- 小项目会显得很重
- 异步和副作用处理需要额外设计

## 四、为什么 Redux 在 Flutter 里不再是默认热门

不是因为 Redux 错了，而是因为：

- Flutter 场景下局部状态很多
- UI 更新频率高
- 团队更倾向更轻量、更贴近 Widget 树的方案

所以现在 Redux 更常作为“架构思想”和“面试题”出现，而不是默认首选。

## 五、Redux 的真正价值

### 1. 可预测性

状态变化只能通过 action -> reducer 完成，路径单一。

### 2. 可追踪性

每一次变更都可以记录：

- 谁触发了变更
- 触发了什么 action
- 状态前后差异是什么

### 3. 易调试

理论上支持：

- action 日志
- 时间旅行调试
- 状态快照回放

这类能力在复杂后台、表单系统、流程系统里价值很高。

## 六、Redux 中最容易答浅的点

很多人会把 Redux 说成：

- “全局状态管理”
- “用 action 改状态”

这太浅了。

更完整的理解是：

Redux 是通过“不可变状态 + 纯函数 reducer + 单向数据流”建立一个高可追踪的状态系统。

## 七、异步和副作用怎么处理

Reducer 必须是纯函数，所以异步请求、日志、埋点、导航等副作用不能直接写进 reducer。

常见做法是中间件：

- thunk
- saga
- custom middleware

在 Flutter 面试里，不一定要背具体库，但要明确一点：

Redux 本身只定义核心状态流，副作用要靠中间层处理。

## 八、Redux 为什么强调不可变状态

原因有三个：

1. 方便比较前后状态差异
2. 避免隐藏副作用
3. 方便调试、回放和撤销

如果直接改原对象：

- 变更来源会变模糊
- debug 成本高
- UI 更新判断更复杂

## 九、Redux 适合什么场景

适合：

- 流程复杂的后台系统
- 可追踪要求高的系统
- 需要严格审计状态变化的业务

不太适合：

- 页面驱动为主的中小型 App
- 状态局部化明显的产品
- 团队不愿承受模板代码成本的项目

## 十、Redux 与 Flutter 的结合思路

如果在 Flutter 中使用 Redux，通常会这样做：

1. Store 保存应用级状态
2. Widget 订阅状态片段
3. 用户交互 dispatch action
4. reducer 产出新状态
5. 需要异步时通过 middleware 处理

核心问题在于：

如果状态树过大、切片不合理，就容易导致订阅范围过宽，UI 更新成本上升。

## 十一、Redux 的常见问题

### 1. 模板代码过多

每个模块都要写：

- action
- reducer
- state
- middleware

小项目很容易觉得“为了规范而规范”。

### 2. 全局状态膨胀

不是所有状态都应该放全局 store，例如：

- 输入框临时值
- 页面局部动画状态
- 控件展开收起状态

这类状态硬放 Redux，收益很低。

### 3. 异步链路理解成本高

如果 middleware 设计不清楚，异步状态流会变复杂。

## 十二、Redux 和 Bloc / Riverpod 的区别

### 对比 Bloc

- Redux：全局状态树更强，单向流更严格
- Bloc：更贴近模块或页面状态流，事件-状态边界清晰

### 对比 Riverpod

- Redux：更强调集中式、可追踪、统一状态流
- Riverpod：更强调依赖管理、组合能力和局部状态灵活性

### 对比 Provider

- Redux：体系完整但偏重
- Provider：轻量，但对状态流约束远弱于 Redux

## 十三、面试高频问答

### 1. Redux 的核心思想是什么？

单一状态树、只读状态、通过纯函数 reducer 根据 action 生成新状态。

### 2. 为什么 reducer 必须是纯函数？

为了保证状态变更可预测、可测试、可回放，避免隐藏副作用。

### 3. 为什么 Redux 不适合所有 Flutter 项目？

因为很多 Flutter 项目页面局部状态多，如果全部集中到 Redux，模板成本和理解成本都偏高。

### 4. Redux 的最大优势是什么？

不是“统一全局状态”，而是“状态变化链路严格可追踪”。

### 5. Redux 的最大代价是什么？

模板代码多、心智负担高、小项目收益不一定明显。

## 十四、面试表达建议

如果面试官问“你怎么看 Redux”，更成熟的回答可以是：

Redux 适合对状态可追踪性要求非常高的场景，它的优势是单向数据流和 reducer 纯函数带来的可预测性，但在 Flutter 的很多页面型业务里会显得偏重，所以我会根据项目规模和团队协作成本判断，而不是默认采用。

这种回答比“Redux 太麻烦了，不如别用”更专业。
