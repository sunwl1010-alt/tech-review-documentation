# Flutter 网络层进阶：Dio、拦截器与治理

## 一、为什么网络层会被问深

初级网络题通常只问：

- 怎么发 GET / POST
- 怎么解析 JSON

但中高级面试和真实项目更关心的是：

1. 为什么选 Dio
2. 拦截器怎么设计
3. token 失效怎么处理
4. 超时、重试、取消、缓存怎么做
5. 错误如何分层
6. 网络层和业务层如何解耦

## 二、为什么多数项目会选 Dio

`http` 包适合简单场景，但中大型项目更常用 Dio。

原因不是“它更高级”，而是它更适合工程化。

### Dio 的核心能力

1. 拦截器
2. 请求取消
3. 超时控制
4. 文件上传下载
5. 统一配置
6. 错误模型相对集中

## 三、网络层不应该只是一堆 `dio.get`

这是常见误区。

一个成熟网络层通常至少要解决：

1. base url 管理
2. header 管理
3. 认证注入
4. 错误映射
5. 日志输出
6. 重试策略
7. 响应解析

否则页面层会越来越重。

## 四、拦截器的职责边界

拦截器很重要，但不能把所有业务逻辑都塞进去。

### 适合放进拦截器的内容

1. 公共 header
2. token 注入
3. 请求日志
4. 响应日志
5. 错误预处理
6. 统一 traceId

### 不适合放进拦截器的内容

1. 页面跳转
2. 复杂业务判断
3. 页面级 toast 文案拼装

### 原则

拦截器应该偏基础设施，不应该变成业务总控。

## 五、token 刷新怎么处理

这是高频面试题。

常见场景：

1. 请求带 access token
2. 服务端返回 401
3. 尝试刷新 token
4. 刷新成功后重放原请求
5. 刷新失败则登出

### 关键难点

不是流程本身，而是并发请求下的竞争问题。

### 常见坑

多个请求同时 401 时：

- 不应该每个请求都单独刷新 token
- 否则会造成重复刷新、状态错乱

### 更稳的思路

- 刷新动作串行化
- 其他请求等待刷新结果
- 刷新成功后统一重放

## 六、超时、取消和重试

### 超时

至少要区分：

1. connect timeout
2. send timeout
3. receive timeout

不要只有一个笼统“请求超时”。

### 取消

适合场景：

- 页面退出
- 搜索联想新请求覆盖旧请求
- 用户主动中断操作

### 重试

不是所有错误都该重试。

适合重试：

- 网络抖动
- 临时网关错误
- 幂等请求失败

不适合重试：

- 参数错误
- 权限错误
- 明确的业务失败

## 七、错误分层应该怎么做

网络层最容易做坏的地方就是错误全混在一起。

推荐至少区分：

1. 网络不可达
2. 超时
3. 服务端错误
4. 认证错误
5. 解析错误
6. 未知错误

### 为什么必须分层

因为：

- 用户提示不同
- 恢复策略不同
- 日志定位方式不同

## 八、日志和调试信息怎么做

网络日志不是简单 `print(response.data)`。

更好的日志体系应该记录：

1. 请求方法
2. URL
3. header
4. body
5. 响应码
6. traceId
7. 耗时

### 注意

生产环境不要直接打印敏感信息：

- token
- 手机号
- 身份证号
- 用户隐私数据

## 九、上传下载为什么容易出问题

因为上传下载通常比普通 JSON 请求复杂：

1. 文件大
2. 耗时长
3. 需要进度监听
4. 失败恢复更复杂

### 关键点

- 要支持取消
- 要支持超时
- 要考虑断点续传或重试策略

## 十、缓存不应该只靠本地存储

缓存不是“把响应存一下”这么简单。

至少要想清楚：

1. 什么数据可以缓存
2. 缓存多久
3. 是否允许脏读
4. 何时主动失效
5. 网络失败时是否回退缓存

这也是为什么网络层通常会和 Repository 层配合。

## 十一、网络层与业务层怎么解耦

比较稳的方式是：

1. 网络客户端只负责请求和响应
2. datasource 负责接口调用与 DTO 转换
3. repository 负责业务数据编排

这样做的好处：

- 页面不依赖底层请求细节
- 缓存逻辑不污染 UI
- 网络实现可以替换

## 十二、面试高频问答

### 1. Dio 相比 `http` 有什么优势？

它更适合工程化，支持拦截器、取消、超时、统一配置和更完整的请求治理能力。

### 2. 拦截器里适合做什么？

适合做公共 header、认证注入、日志和错误预处理，不适合堆复杂业务逻辑。

### 3. token 刷新最难的点是什么？

不是刷新本身，而是并发请求下的串行化控制和失败兜底。

### 4. 什么错误适合自动重试？

临时网络问题和幂等请求失败适合；权限、参数、业务错误通常不适合。

## 十三、推荐回答模板

如果面试官问“你怎么设计 Flutter 网络层”，更完整的回答方式是：

1. 用 Dio 做统一客户端
2. 用拦截器处理认证、日志和错误预处理
3. 把错误映射成统一模型
4. 把缓存、重试、重放和数据编排放到 datasource / repository 层
5. 页面只消费最终状态，不直接处理底层请求细节

这样回答会明显比“我会用 Dio 发请求”成熟。
