# Flutter 数据层架构：Repository、缓存与协作

## 一、为什么数据层常被忽视

很多 Flutter 项目网络能通、页面能显示，但一到复杂场景就开始乱：

1. 页面直接请求接口
2. 缓存逻辑散在各处
3. 本地数据和远端数据冲突
4. 错误和空态处理不统一

这不是“代码不够多”，而是数据层没有设计。

## 二、Repository 到底解决什么问题

Repository 不是“多包一层”。

它的核心价值是：

1. 隔离数据来源
2. 屏蔽底层实现细节
3. 编排远端和本地数据
4. 给上层提供稳定接口

### 上层关心什么

- 我要拿到用户资料
- 我要更新购物车
- 我要加载详情页数据

### 上层不应该关心什么

- 这次是网络来的还是缓存来的
- DTO 怎么转实体
- 本地表结构是什么

## 三、推荐的数据层角色划分

### 1. remote datasource

负责：

- 调接口
- 处理请求参数
- 返回 DTO

### 2. local datasource

负责：

- 本地缓存读写
- 数据库存取
- key-value 存储

### 3. repository

负责：

- 组合 remote 和 local
- 统一错误输出
- 转换为领域模型
- 决定缓存策略

## 四、为什么页面不应该直接调接口

因为那会导致：

1. UI 和数据源强耦合
2. 测试困难
3. 缓存难接入
4. 业务逻辑下沉到页面

页面应该依赖 repository 或 use case，而不是直接依赖 Dio。

## 五、DTO、Entity、UI Model 要不要分

这个问题没有唯一答案，但中大型项目最好区分：

### DTO

对应接口返回结构。

### Entity / Domain Model

表达业务语义，尽量不依赖接口字段命名细节。

### UI Model

在必要时做页面展示层映射。

### 为什么值得分

因为接口结构经常变化，但业务语义和 UI 不一定应该跟着一起碎裂。

## 六、缓存策略是数据层核心能力

缓存不是附属品，很多时候它决定用户体验。

### 常见缓存策略

1. cache aside
2. network first
3. cache first
4. stale-while-revalidate

### 什么时候用 `network first`

适合：

- 实时性更高的数据

### 什么时候用 `cache first`

适合：

- 配置、字典、低频变化数据

### 什么时候用 `stale-while-revalidate`

适合：

- 列表页
- 首页内容
- 用户可接受短暂旧数据的场景

## 七、离线和弱网怎么处理

数据层设计成熟与否，弱网场景最能看出来。

### 常见策略

1. 请求失败回退缓存
2. 写操作做本地队列或草稿
3. 明确展示“旧数据”状态

### 不要做的事

网络一失败就直接空白页，且不区分“无数据”和“加载失败”。

## 八、数据一致性问题

缓存一旦存在，就有一致性问题。

你至少要考虑：

1. 本地缓存何时失效
2. 提交更新后是否立即刷新缓存
3. 多页面数据是否同步
4. 乐观更新失败怎么回滚

## 九、乐观更新与保守更新

### 乐观更新

先更新 UI，再回写服务端。

优点：

- 交互更流畅

风险：

- 失败后需要回滚

### 保守更新

服务端成功后再更新 UI。

优点：

- 一致性简单

缺点：

- 交互反馈偏慢

## 十、分页和列表数据的设计

分页不是“多传一个 page 参数”那么简单。

要考虑：

1. 首次加载
2. 下拉刷新
3. 上拉加载更多
4. 去重
5. 结束判断

### 常见坑

- 重复数据
- 并发加载多页
- 页码状态错乱
- 刷新后旧数据混入

## 十一、Repository 和状态管理如何协作

比较稳的链路通常是：

1. UI 触发事件
2. 状态层调用 repository
3. repository 选择 remote/local 策略
4. 返回统一结果
5. 状态层把结果转换成页面状态

### 原则

repository 不负责页面 loading 样式；
状态层不负责底层缓存细节。

## 十二、测试为什么依赖 Repository

Repository 的一个实际价值是方便测试。

因为你可以：

1. mock repository
2. mock datasource
3. 验证缓存和网络协作逻辑

如果页面直接依赖 Dio，测试会很痛苦。

## 十三、面试高频问答

### 1. Repository 模式的核心价值是什么？

隔离数据来源、统一数据访问入口、隐藏实现细节、承接缓存和业务数据编排。

### 2. 页面为什么不应该直接请求接口？

因为那会让 UI 和底层实现耦合，难测试、难缓存、难扩展。

### 3. 缓存策略怎么选？

要看数据实时性、用户体验要求和一致性要求，而不是全都一刀切。

### 4. DTO 和 Entity 为什么要分？

因为接口结构和业务语义不是一回事，分层后更抗变化。

## 十四、推荐回答模板

如果面试官问“Flutter 数据层怎么设计”，比较成熟的回答方式是：

1. 用 remote/local datasource 分离远端和本地数据源
2. 用 repository 统一对上层暴露接口
3. 在 repository 中做缓存策略、错误映射和数据编排
4. 状态层只处理业务状态，不处理底层请求细节

这样回答会明显比“我会写 repository”更完整。
