# Android 测试体系专题（单测 / UI / 集成）

## 1. 结论先说

高质量 Android 测试不是“覆盖率数字”，而是分层验证关键风险：
- 单元测试：快、稳定，验证业务规则
- UI 测试：验证关键交互链路
- 集成测试：验证模块协作与真实依赖行为

## 2. 测试金字塔（推荐）

- 底层：Unit Test（最多）
- 中层：Integration Test
- 顶层：E2E/UI Test（最少但关键）

原因：UI 测试成本高、脆弱性高，不应成为主要测试手段。

## 3. 单元测试重点

- ViewModel 状态流转（loading/success/error）
- UseCase 业务分支
- Repository 错误映射

常用工具：JUnit、MockK/Mockito、Turbine（Flow）。

## 4. UI 测试重点

- 登录、支付、下单等关键主路径
- 权限弹窗与异常降级路径
- Compose 场景用 `compose-ui-test`

避免：对纯样式细节做大量脆弱断言。

## 5. 集成测试重点

- Room migration test
- 网络层 MockWebServer（超时/401/5xx）
- WorkManager 任务状态和重试路径

## 6. 测试数据与环境治理

- 使用可复用 fixture，避免魔法常量
- 测试隔离：每个 case 独立数据
- CI 并行执行，分层报告失败原因

## 7. 常见追问

### Q1：为什么覆盖率高仍然会有线上事故？
答：覆盖率只说明“跑到了”，不代表断言质量和风险覆盖完整。

### Q2：如何选“必须写 UI 测试”的场景？
答：用户价值高、回归代价高、历史故障高发的关键路径。

### Q3：如何保证测试在 CI 稳定？
答：隔离外部依赖、控制时钟/线程、减少 flaky UI 断言。

## 8. 60秒答题模板

- 结论：测试要分层，单测保规则，集成保协作，UI 保关键路径。
- 依据：成本、稳定性、反馈速度不同，不能只靠一种测试。
- 落地：我会先补高风险模块单测，再补关键链路 UI/集成测试。
- 边界：追求覆盖率数字而忽略断言质量没有实际价值。
